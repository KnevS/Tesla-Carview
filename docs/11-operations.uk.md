## 💾 Резервне копіювання та відновлення

### Створити резервну копію

**Через веб-інтерфейс (рекомендовано):**

1. Увійти як адмін → **Admin → Керування даними**
2. Картка "💾 Повна резервна копія та відновлення" → кнопка **"⬇ Backup erstellen"**
3. Завантажується JSON-файл — ім'я файлу `tesla-carview-backup-<slug>-YYYY-MM-DD.json`

Вміст: усі 26 таблиць активної tenant DB (авто, поїздки + GPS-точки, сесії зарядки, телеметрія, журнал поїздок, інтервали обслуговування, користувачі, passkey-credentials, audit-журнали, налаштування, Tesla OAuth-токени, Virtual Key, юридичні погодження, історія змін поїздок). Свідомо виключено: `push_subscriptions` (специфічно для браузера) та `refresh_tokens` (вони живуть у `master.db`).

> **Passkeys**: `passkey_credentials` включено в резервну копію. Після відновлення на **той самий сервер** зареєстровані passkeys працюють одразу — `credential_id` зберігається на стороні сервера, а `user_id` зберігається при відновленні. Відновлення на інший сервер чи домен потребує переєстрації passkeys (WebAuthn прив'язаний до домену).

**Через CLI / cron** (для зовнішніх стратегій резервування):

```bash
# Резервує файли SQLite напряму — атомарно, без зупинки сервісу
docker compose -f docker-compose.prod.yml exec backend sh -c \
  "sqlite3 /app/data/master.db '.backup /app/data/backup-$(date +%F).db'"
docker cp tesla-carview-backend:/app/data/. /path/to/backup/
```

Рекомендовано: також зберігати UI-резервну копію на зовнішньому диску — один JSON-файл per tenant є портативним і версіонованим.

### Відновити з резервної копії

**Сценарій:** встановлено нову систему, або стара заплуталась. Повернути попередній стан:

1. Принаймні один раз пройдіть майстер налаштування (створіть обліковий запис адміна)
2. Увійти як адмін → **Admin → Керування даними → "⬆ Backup wiederherstellen…"**
3. Обрати JSON-файл + ввести підтвердження `WIEDERHERSTELLEN`
4. "Jetzt wiederherstellen" → сервер спершу створює **захисну резервну копію поточної `.db`** (шлях повертається у повідомленні про успіх), потім очищає всі таблиці та повторно наповнює їх з JSON, усе всередині **однієї транзакції**, що відкочується при помилці
5. Вийти + увійти знову, готово

### Шари безпеки при відновленні

- middleware `requireAdmin`
- Фразу підтвердження `WIEDERHERSTELLEN` потрібно ввести точно
- Pre-restore резервна копія на рівні файлу (`<dbname>_pre_restore_<timestamp>.db`)
- Перетин колонок: коли в live-схемі є перейменована колонка, ця колонка пропускається замість того, щоб вбити весь імпорт
- Запис в audit-журнал для кожної резервної копії та кожного відновлення

## 🌙 Нічне обслуговування

Запускається щодня між **03:30 та 03:40 Europe/Berlin** (захищено від DST через `Intl.DateTimeFormat`). Зупиняється при кожному перезапуску backend, повертається з backoff 2 хв.

### Що воно робить

| Де | Завдання |
|---|---|
| `master.db` | Видалити прострочені `refresh_tokens` |
| `master.db` | Видалити стани `oauth_pkce` старші за 24 год |
| `master.db` | Видалити soft-revoked tenant-invites старші за 30 днів |
| `master.db` | `VACUUM` + `wal_checkpoint(TRUNCATE)` |
| кожен `tenant.db` | Видалити `audit_logs` старші за 180 днів |
| кожен `tenant.db` | Видалити використані/прострочені `user_invites` старші за 30 днів |
| кожен `tenant.db` | `wal_checkpoint(TRUNCATE)` |
| кожен `tenant.db` | `VACUUM` лише коли DB > 50 MB |
| кожен `tenant.db` | Audit-запис `system_maintenance` з підрахунками |

### Запустити вручну

**UI:** Система → Стан системи → "🌙 Нічне обслуговування" → **"Запустити зараз"**.

**API:**
```bash
curl -X POST https://carview.example.com/api/system/maintenance-now \
  -H "Authorization: Bearer $ADMIN_JWT"
```

### Перевірити останній запуск

```bash
curl https://carview.example.com/api/system/maintenance-log \
  -H "Authorization: Bearer $ADMIN_JWT" | jq
```

Показує останні до 50 запусків з підрахунками, тривалістю та статусом помилок.

## ⬆️ Авто-оновлення (opt-in)

> ⚠️ **Вимкнено за замовчуванням.** Увімкнення означає, що ваша система щонічно отримує нові коміти з `main` і перезбирає контейнер. Спершу переконайтеся, що `deploy/update.sh` працює чисто на вашому setup.

### Увімкнення

```bash
# backend/.env
AUTO_UPDATE_ENABLED=true
UPDATE_REPO_DIR=/opt/tesla-carview   # за замовчуванням саме це, перевизначте лише якщо інше
```

Потім перезапустіть backend:
```bash
docker compose -f docker-compose.prod.yml up -d --build backend
```

### Що відбувається вночі

1. `git fetch origin main` у налаштованому шляху до репо
2. Порівнює `git rev-parse HEAD` з `origin/main`
3. Якщо різниться: `bash deploy/update.sh` (timeout 10 хв)
4. Під час перезбірки frontend автоматично показує **maintenance-overlay** (див. `frontend/src/components/MaintenanceOverlay.vue`) з жартами про Tesla — користувачі майже не помічають
5. Статус (локальний hash, віддалений hash, результат оновлення) потрапляє в maintenance-журнал

### Ручне оновлення в будь-який час

```bash
cd /opt/tesla-carview
bash deploy/update.sh
```

## 🧪 Демо-режим

Для **тестерів без Tesla**. Якщо ви запускаєте це лише для себе, залиште вимкненим.

### Увімкнення

```bash
# backend/.env
DEMO_ENABLED=true
```

Перезапустіть backend. Додатковий tenant зі slug `demo` і файлом DB `data/tenants/<uuid>.db` створюється при першому старті.

### Жорсткі ліміти (закодовані в `routes/demo.js`)

| Константа | За замовчуванням | ENV-змінна | Значення |
|---|---|---|---|
| `MAX_ACTIVE_DEMO_USERS` | `200` | `MAX_ACTIVE_DEMO_USERS` | Одночасні тестери. HTTP 503 при заповненні. |
| `DEMO_SIGNUPS_PER_IP` | `2` / 24 год | `DEMO_SIGNUPS_PER_IP` | Максимум 2 реєстрації per IP за 24-годинне вікно |
| `DEMO_LIFETIME_DAYS` | `2` | `DEMO_LIFETIME_DAYS` | Обліковий запис + усі його дані видаляються через 2 дні, без залишків |

Усі три можна перевизначити через змінну середовища — для приватного інстансу з `DEMO_ENABLED=true` розгляньте `MAX_ACTIVE_DEMO_USERS=5` і `DEMO_LIFETIME_DAYS=1`.

### Що бачать тестери

- Сторінка входу показує синю картку "🧪 Tesla Carview ausprobieren — ohne Tesla" з вільними слотами
- Один клік → створюється користувач `tester-<hex>`, виконується вхід, засівається фейкове авто + 3 тижні історії
- Банер угорі застосунку: "Demo-Modus — Konto und Daten werden am DD.MM.YYYY automatisch gelöscht (X Tage übrig)"
- Сторінки приватності та умов автоматично показують **доповнення для тестерів** (без SLA, без підтримки, фейкові дані, видалення через `DEMO_LIFETIME_DAYS` днів)
- Кожні 30 хв: нова фейкова поїздка per demo-авто — щоб демо здавалося живим

### Очищення

- Кожні 6 годин запускається demo lifecycle: користувачі з `expires_at < now` видаляються в одній транзакції разом з усіма залежними таблицями (авто, поїздки, GPS-точки, зарядка, батарея, телеметрія, журнал поїздок, MFA-коди, audit-журнали, локації зарядки, інтервали обслуговування)
- Сам demo-tenant залишається — стираються лише дані тестера
- **Ізоляція**: demo-slug **ніколи** не записується в `localStorage` — тестер, який закриває вкладку браузера і знову відкриває продакшн-URL, не потрапить помилково в demo-tenant

## 🛡️ Моніторинг та самовідновлення

Cron-завдання (`/opt/monitoring/bin/heal.sh`) запускається кожні 15 хвилин і спостерігає за основними сервісами:

1. **Статус контейнера** — перевіряє `docker inspect` для `tesla-carview-backend`, `-frontend` і `-nginx`; якщо контейнер не в стані `running`, він перезапускається через `docker compose up -d <service>`.
2. **Health endpoint** — коли всі контейнери запущені, перевіряється `GET /api/health`; якщо відповідь не `{"status":"ok"}`, контейнер backend перезапускається.
3. **Email-сповіщення** — після кожного автоматичного перезапуску на налаштовану адресу надсилається email-сповіщення (якщо встановлено).
4. **Ротація журналів** — `/var/log/tcv-heal.log` автоматично обрізається до останніх 500 рядків при перевищенні 1 MB.

**Конфігурація** (Admin → Система → Моніторинг та самовідновлення):

| Налаштування | Опис |
|---|---|
| Самовідновлення увімк/вимк | Ключ DB `monitoring.heal_enabled`; встановіть `false`, і cron-завдання негайно завершиться |
| Email для сповіщень | Ключ DB `monitoring.alert_email`; порожній = без сповіщень |

**API endpoints** (лише адмін):
- `GET /api/system/monitoring-config` — читає поточну конфігурацію
- `PUT /api/system/monitoring-config` — зберігає конфігурацію
- `GET /api/system/monitoring-log?lines=50` — повертає останні N рядків з heal- та security-журналів

**Перевірити журнали вручну:**
```bash
tail -50 /var/log/tcv-heal.log
tail -50 /var/log/security-check.log
```

**Cron-запис** (`/etc/cron.d/tesla-carview-monitoring`):
```
*/15 * * * * root /opt/monitoring/bin/heal.sh >/dev/null 2>&1
```

## 📊 Стан системи з одного погляду

UI: **Admin → Система** → адмін бачить кольорову картку-світлофор зверху. Backend endpoint: `GET /api/system/health` (лише адмін). Перевірки:

| Перевірка | Зелений | Жовтий | Червоний | Info (приглушено) |
|---|---|---|---|---|
| Tesla OAuth token | дійсний, > 7 днів залишилось | < 7 днів залишилось | прострочений або відсутній | — |
| Virtual Key | створений | — | не створений | — |
| Fleet Telemetry | остання точка даних < 24 год | < 7 днів | нічого або > 7 днів | — |
| Tesla poller | останнє опитування < 60 хв | < 1 день | — | — |
| Tenant DB | інформативно — розмір у MB | — | — | — |
| Нічне обслуговування | останній запуск < 25 год | — | — | — |
| OpenChargeMap | live-перевірка OK | — | перевірка не вдалася (ключ встановлено) | ключ не налаштовано |
| HERE Maps | live-перевірка OK | — | перевірка не вдалася (ключ встановлено) | ключ не налаштовано |

Опціональні сервіси (OCM, HERE) рахуються як помилки лише коли ключ налаштовано, але endpoint не відповідає. Без ключа: статус `info`, приглушено, без впливу на загальний колір світлофора.

## 🔍 Перегляд журналів

**Журнали контейнерів:**
```bash
docker compose -f docker-compose.prod.yml logs -f --tail 200 backend
```

**Audit-журнал** (події, релевантні для безпеки, per tenant):
- UI: **Admin → Audit log** з фільтрами (action, user-id, дата) та експортом у CSV
- API: `GET /api/audit` (лише адмін)

**Maintenance-журнал** (останні нічні запуски):
- UI: Система → "🌙 Нічне обслуговування" → деталі
- API: `GET /api/system/maintenance-log` (лише адмін)

## 🚨 Надзвичайна ситуація: скидання бази даних

Коли все горить і допоможе лише чистий перезапуск:

```bash
# 1. СПЕРШУ зробіть резервну копію (див. вище)
# 2. Зупинити контейнери
docker compose -f docker-compose.prod.yml down

# 3. Витерти data-директорію — УСІ ДАНІ ЗНИКНУТЬ
# Дані живуть у bind-mount ./data (не в Docker named volume!)
rm -rf ./data/master.db ./data/tenants/

# 4. Запустити з нуля — майстер налаштування з'явиться автоматично
docker compose -f docker-compose.prod.yml up -d
```

> З v2.0 SQLite-бази живуть під `./data` як bind-mount (відносно Compose-файлу), **не** в Docker named volume. `docker volume rm` не впливає на цей setup.

Щоб відновити резервну копію після цього, пройдіть майстер налаштування з тимчасовим обліковим записом адміна, увійдіть і скористайтеся UI-процесом відновлення.
