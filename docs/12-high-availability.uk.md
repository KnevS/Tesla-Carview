# 🛡️ High Availability (HA) — варіанти архітектури

> 🤖 *Цей український переклад створений з AI-підтримкою з [README.en.md](README.en.md). Виправлення приймаються на GitHub.*

> 🇩🇪 [Auf Deutsch lesen](12-high-availability.md) · 🇬🇧 [Read in English](12-high-availability.en.md) · 🏠 [Індекс документації](.)

Tesla Carview постачається як **single-node розгортання** на одному сервері Linux або Raspberry Pi — повністю достатньо для самохостингу домогосподарства: швидко, легко, дешево. Для **комерційних multi-vehicle / multi-tenant інсталяцій** з визначеними вимогами SLA доступне на запит **high-availability (HA) налаштування** — проєктується індивідуально, не є частиною стандартного репо.

---

## Коли HA виправдане?

Орієнтовні правила. Якщо щось з цього "так" — обговорення HA має сенс:

- 🚛 **Понад 5 авто** та/або кілька tenants у продакшні → один збій одразу б'є по багатьох користувачах
- 📑 **Податково-обов'язковий журнал поїздок** (BMF-експорт) — втрата даних швидко стає дорогою
- ⏱ **Активні команди Tesla** (preconditioning, графік зарядки) мають працювати 24/7
- 🛠 **Maintenance-вікна лише поза німецькими робочими годинами**, без ризику вночі
- 📈 **Рівні сервісу, обіцяні третім сторонам** (оператори флоту, корпоративні автопарки)

Для приватного використання з 1–2 авто single-node + щоденна резервна копія + процес відновлення цілком достатньо (див. [11-operations.en.md](11-operations.en.md)).

## Можливі HA-топології (огляд)

Ці варіанти реалізовні і можуть бути поставлені на запит — правильний вибір залежить від цілей доступності, бюджету та наявної інфраструктури.

### Tier 1 — Warm standby (RTO ≈ 5 хв, RPO ≈ 5 хв)
- Другий ідентичний сервер на другому майданчику
- Періодична реплікація резервної копії SQLite (наприклад, через `litestream` → S3-сумісне об'єктне сховище)
- DNS failover або floating IP для перемикання
- **Плюс:** дешево, просто, покриває 95 % сценаріїв збоїв
- **Застереження:** можлива коротка втрата даних (від секунд до кількох хвилин)

### Tier 2 — Active-active за load balancer (RTO < 1 хв, RPO ≈ 0)
- Кілька backend-контейнерів за L7 load balancer (nginx/HAProxy/Traefik)
- SQLite замінено на server-side Postgres (або sqlite-сумісний кластер на кшталт `rqlite`)
- Stateless backend (JWT-based, session affinity не потрібен)
- Спільне об'єктне сховище для резервних копій і міграцій tenant
- **Плюс:** реальний failover без втручання людини
- **Застереження:** більше рухомих частин, вищі операційні витрати

### Tier 3 — Гео-резервований (RTO < 1 хв, RPO ≈ 0, покриває регіональні збої)
- Tier 2 розгорнутий у другому географічному регіоні (наприклад, Frankfurt + Berlin)
- База даних із синхронною реплікацією або quorum-кластер (PostgreSQL Patroni, CockroachDB)
- Anycast DNS або global load balancer для міжрегіональної маршрутизації
- **Плюс:** переживає збій ЦОД
- **Застереження:** вимоглива конфігурація; затримку до бази через регіони треба планувати

### Tier 4 — Kubernetes-нативне multi-replica розгортання
- Helm chart для backend + frontend + reverse proxy
- Horizontal Pod Autoscaler (HPA)
- PersistentVolume з реплікованим backend storage class (Longhorn, OpenEBS, Ceph)
- Sealed Secrets / External Secrets Operator для значень `.env`
- **Плюс:** інтегрується з існуючою cloud / on-prem K8s-платформою
- **Застереження:** виправдане лише при ≥ 10 tenants або при наявному K8s-ландшафті

## Що вже HA-ready у стандартному репо

Навіть без явного HA-налаштування архітектура спроєктована так, щоб апгрейд не ламав усе:

- **Stateless backend** — session store не потрібен, горизонтальне масштабування прямолінійне (JWT + httpOnly refresh cookie)
- **Multi-tenant ізоляція** — кожна tenant DB окрема, per-tenant резервне копіювання/відновлення через `/api/data/backup` + `/restore` (див. [11-operations.en.md](11-operations.en.md))
- **Maintenance-overlay у frontend** — користувачі не бачать "помилки" під час перемикання контейнера, вони бачать дружню картку оновлення
- **Audit-журнал** — кожна релевантна для безпеки дія записується, криміналістичний слід після failover
- **Нічне обслуговування** — DB vacuum, прострочені токени, старі audit-журнали очищаються автоматично
- **System-health endpoint** (`/api/system/health`) — безпосередньо придатний як Kubernetes liveness/readiness probe або у зовнішньому моніторингу (Uptime Kuma, Healthchecks.io, Statping)

## Як діяти, якщо вам потрібен HA

1. **Уточнити вимоги** — цілі RTO/RPO, кількість tenants та авто, compliance (GDPR processing agreement, BMF retention)
2. **Вибрати топологію** — один з tier вище або проєктно-специфічна суміш
3. **План міграції** — single-node → HA без втрати даних, можливо, спершу перехід на Postgres
4. **Передача експлуатації** — runbooks, моніторинг, on-call setup, incident playbooks
5. **Навантажувальні тести** — failover-навчання, репетиції backup-restore

Ці кроки плануються індивідуально з оператором у клієнтському проєкті — без one-size-fits-all рецепту в репо.
